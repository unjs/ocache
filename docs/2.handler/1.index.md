# Caching HTTP Handlers

Wrap HTTP handlers with `defineCachedHandler` for automatic response caching with `etag`, `last-modified`, and `304 Not Modified` support. It works with any runtime that has standard `Request`/`Response` — the event is just `{ req: Request; url?: URL }`.

```ts
import { defineCachedHandler } from "ocache";

const handler = defineCachedHandler(
  async (event) => {
    // event.req is a standard Request object
    const url = event.url ?? new URL(event.req.url);
    const data = await getExpensiveData(url.pathname);
    return new Response(JSON.stringify(data), {
      headers: { "content-type": "application/json" },
    });
  },
  {
    maxAge: 300, // Cache for 5 minutes
    swr: true,
    staleMaxAge: 600,
    varies: ["accept-language"], // Vary cache key by these headers (also emitted as `Vary`)
    allowQuery: ["color"], // Vary cache by these query params only
  },
);
```

## What it does

- **Auto-generates cache keys** from the URL path and variable headers.
- **Conditional requests** — handles `304 Not Modified` via `if-none-match` / `if-modified-since`.
- **Response headers** — sets `cache-control`, `etag`, and `last-modified`, but never clobbers an explicit `cache-control` set by the handler. SWR / `s-maxage` / `max-age` directives are only synthesized when the handler didn't set one.
- **`Vary`** — emits a `Vary` response header from `opts.varies`, merging with any `Vary` the handler already set (case-insensitive dedup) so downstream caches store per-variant.
- **Binary responses** — images, protobuf/MVT tiles, and other non-text bodies are stored losslessly and replayed byte-for-byte; text bodies keep stable text etags.

## Header Filtering

Only the headers listed in `varies` participate in the cache key, and non-variable headers are filtered before the handler runs so cache keys stay consistent. This request narrowing (variable-header filtering, cookie stripping, query narrowing) only applies to cacheable calls.

## Bypassing the Cache

Non-`GET`/`HEAD` requests bypass the cache entirely and reach the handler with their request untouched — including headers, cookies, query, and the request body (which the internally rewritten `Request` would otherwise drop). You can compose additional bypass logic via `shouldBypassCache`.

Bypassed responses pass through untouched: no body buffering (so streaming/binary bodies survive), no synthesized cache headers, and no bogus `304` for a non-cacheable method.

## Headers-only Mode

Use `headersOnly` to handle conditional requests without caching the full response:

```ts
const handler = defineCachedHandler(myHandler, {
  headersOnly: true,
  maxAge: 60,
});
```

## Framework Integration Hooks

For adapting to a specific framework's event/response types, `CachedEventHandlerOptions` exposes:

- `toResponse(value, event)` — convert a handler return value to a `Response` (default: plain `Response` constructor).
- `createResponse(body, init)` — create the final `Response` from cached data (default: `new Response()`).
- `handleCacheHeaders(event, conditions)` — custom `304` conditional check (default: built-in `if-none-match` / `if-modified-since`).
